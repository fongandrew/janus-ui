Janus v2 · Design review · Aesthetic gap

The accessibility-led taste in v1 that the v2 spec keeps dropping on the floor.

The v2 spec captures the structure of v1 well — tokens, cascade, objects, components. What it under-specifies is the visual mechanics that make v1 feel calm and legible on any palette. Every gap below is small, but cumulatively they're why v1 reads as designed and early v2 attempts read as generic.

v1 leans hard on a small library of self-tuning recipes: borders that take their color from the surface they sit on, hovers that lean on a single filter instead of a per-palette color, focus rings layered in two passes, link colors that auto-bump their weight to clear the contrast floor. None of these are written down as design rules in the v2 spec — they exist as CSS mixins (t-hover, t-focus-ring, t-border, v-text-muted) that an implementing agent reading PLAN.md doesn't know to look at.

The pattern across all of them is the same: encode the relationship, not the value. v1 doesn't pick a border color per palette — it states, in one line, that every border sits 17.5% of the way from the surface toward the ink, in HSL. The palette can move; the relationship holds. That's the taste. Below is the catalog of relationships v2's spec needs to name.

Each card on this page is itself rendered with the v1 mechanics it documents. Hover the buttons, tab into the links, mouse vs. arrow-key through the menu in §9. The document is the proof.

I

Surface & color

5 principles · maps to v2 spec §7
01

Borders take their color from the surface they sit on.

One color-mix recipe places every border at a fixed perceptual distance from any background — light, dark, branded, danger — without a per-palette knob.

Fixed `rgb(0 0 0 / 17.5%)` border
paper
teal
green
sand

Reads on paper, disappears on dark, fights against tinted surfaces.

`color-mix(black 17.5%, bg)`
paper
teal
green
sand

Same visual weight on every surface; flips base to white in dark mode.

Gap in v2 spec §7 names --v-bg, --v-fg, --v-link, --v-accent, --v-muted — but no border-color knob and no recipe for how borders take their color. Implementers default to a flat rgb(0 0 0 / 12%). 
/* tokens/color.css — add */
--v-border-dynamic-base: light-dark(black, white);
--v-border-dynamic-mix:  light-dark(17.5%, 50%);
--v-border-color: color-mix(in hsl,
  var(--v-border-dynamic-base) var(--v-border-dynamic-mix),
  var(--v-bg));
02

"Flat" pages aren't flat — they carry a faint two-stop tint.

Two low-opacity gradients (warm peach + cool teal in light; deep green + cyan in dark) give the eye somewhere to rest. Read it on this page's background.

Pure `hsl(30 12% 98.5%)`
Looks like a wireframe.
+ 15% peach @ 102°  ·  + 7% teal @ 12°
Looks like a page someone designed.
Gap in v2 spec §5.1: --v-bg defaults to light-dark(white, black). No mention of --v-body-bg as a derived token or of the gradient recipe. 
/* tokens/color.css — add */
--v-body-bg: linear-gradient(102deg, transparent, hsl(30 75% 75% / 15%)),
              linear-gradient(12deg,  transparent, hsl(183 25% 71% / 7%)),
              var(--v-bg);

/* base.css */
body { background: var(--v-body-bg); background-attachment: fixed; }
03

Tooltips are always inverted; popovers always match.

A tooltip floats above the page — its surface should read as a separate layer. A popover replaces UI — its surface should match. v1 names both; v2 collapses them.

Gap in v2 spec §7 lists v-colors-primary/danger/success/warn/info/muted — no v-colors-tooltip, no v-colors-popover. v2 implementers paint tooltips with the body palette. 
/* variants/colors.css — add to the catalog */
.v-colors-tooltip {

04

Code blocks stay dark in light mode.

Code is a terminal-context: monospaced, dense, syntax-colored. v1 keeps pre on an ink surface in light mode so users don't context-switch eye adaptation when scanning code on a paper page.

Inline code uses a tinted paper background and inherits text color. Block code stays dark: 

function deriveBorder(bg) {
  return `color-mix(in hsl, currentColor 17.5%, ${bg})`;
}
Gap in v2 spec §9.5 defines o-code (sizing only). No v-colors-code / v-colors-pre with the asymmetric paper/ink treatment. v2 implementers paint <pre> with the page surface. 
/* variants/colors.css */
.v-colors-code { /* tinted-paper, inline code */
  --v-bg: color-mix(in hsl, var(--v-muted) 8%, var(--v-bg));
}
.v-colors-pre { /* ink, even in light mode */
  --v-bg: #141821;
  --v-fg: #fdf3e1;
  --v-muted: #d1ccc4;
}
code { @apply v-colors-code; }
pre  { @apply v-colors-pre; }
05

Callouts auto-bump every weight by one step.

A tinted background reduces contrast against the page's body weight. v1's v-colors-callout shifts normal→500, label→600, strong→800 for everything inside, so prose stays readable on a mid-tone surface.

no bump
See the spec gap below for the full recipe. Body weight on tinted bg loses contrast.
v1 bump
See the spec gap below for the full recipe. Body weight on tinted bg compensates.
Gap in v2 spec §7.1 introduces per-color --v-*-weight-min floors (link, accent, muted). It doesn't generalize to "tinted-surface variants bump all weights." 
/* variants/colors.css */
.v-colors-callout,
.v-colors-warn,
.v-colors-info {
  --v-bg: color-mix(in hsl, currentcolor 8%, var(--v-bg));
  /* bump the whole stack by ~100 */
  --v-font-weight-normal:   500;
  --v-font-weight-label:    600;
  --v-font-weight-strong:   800;
  font-weight: var(--v-font-weight-normal);
}
II

Hover, press, focus

4 principles · maps to v2 spec gap
06

Hover is one filter, applied to anything.

v1's filter: contrast(0.95) saturate(1.15) works on every button, link, badge, and tone — primary, danger, ghost, paper. No per-palette hover-color knobs. Hover the rows below.

Naïve — per-palette `:hover {background: …}`
Link

Each tone needs its own hover color. Forget one, get a dead state.

v1 — single filter recipe
Link

Try hovering. One rule covers every tone.

Gap in v2 spec §10.1 says "c-button · hover state, outer shadow." It doesn't say what the hover state is. Without guidance, agents reach for per-palette darken/lighten. 
/* tools/states.css */
@define-mixin t-hover {
  filter: contrast(0.95) saturate(1.15);
  will-change: filter;  /* nudges Safari GPU path */
}

/* tools/elements.css */
@define-mixin t-button-style {
  /* … */
  &:hover { @mixin t-hover; box-shadow: var(--v-shadow-md); }
  &:active { box-shadow: none; }
}
07

Focus is two layers: crisp outline + soft halo.

Outline alone is easy to miss against busy layouts; halo alone is fuzzy. v1 stacks box-shadow (ring-alt halo) under outline (ring-crisp). Tab through the buttons.

Outline only
v1 — halo + outline
Gap in v2 spec §5.1 mentions --v-accent "(focus ring)" — one knob. v1 uses two (--v-ring crisp, --v-ring-alt soft) and stacks them. No t-focus-ring spec. 
/* tokens/color.css */
--v-ring:     var(--v-accent);                    /* crisp outline */
--v-ring-alt: color-mix(in hsl, var(--v-accent) 35%, transparent);
--v-shadow-focus: 0 0 0 0.125rem var(--v-ring-alt);

/* tools/states.css */
@define-mixin t-focus-ring {
  box-shadow: var(--v-shadow-focus);
  outline: 2px solid var(--v-ring);
  outline-offset: 2px;
}
08

Inline links flip to a double underline on focus.

A box-outline around an inline link breaks the reading line. v1's link focus changes text-decoration-style to double — strongly visible, doesn't disturb the run of text. Tab through the paragraph.

Outline focus on inline link

A paragraph with an inline link that gets a box outline. The box bursts above and below the text baseline and disrupts the reading line.

v1 — double underline

A paragraph with an inline link that gets a double underline. The text flow is preserved; the doubling is unmistakable up close and at a glance.

Gap in v2 spec The link / focus story in v2 is missing entirely. base.css in PLAN.md phase 1.5 lists "links" but doesn't specify the :focus-visible treatment. 
/* base.css */
a:where([href]) {
  color: var(--v-link);
  font-weight: var(--v-link-weight-min, inherit);
  text-decoration: underline;
  text-underline-offset: 0.25rem;
}
a:focus-visible {
  outline: none;
  text-decoration-style: double;
}
09

Menu highlight follows mouse or keyboard — not both.

When the user is mousing, hover highlights. When they're arrow-keying through a menu, the active descendant highlights regardless of cursor position. v1 toggles body[data-v-kb-nav="true"] from script.

Open file
Save as…
Recent files
Preferences
Quit

In mouse mode, hover the rows. In keyboard mode, Save as… stays highlighted (the "active descendant") regardless of hover.

Gap in v2 spec PLAN.md phase 5 lists t-active-descendant and t-roving-focus as handler modules but doesn't mention the page-level data-v-kb-nav flag that makes the CSS conditional. 
/* page-level state driven by dom/handlers/kb-nav.ts */
body[data-v-kb-nav="true"] { /* arrow-key mode */ }

/* components/menu.css */
body:not([data-v-kb-nav="true"]) .o-menu-item:hover,
body[data-v-kb-nav="true"] .o-menu-item[data-active] {
  @apply v-colors-highlight;
}

Type, weight, and stroke

3 principles · maps to v2 spec §5 / §7.1
10

Hue-distinguished text bumps its weight to clear the contrast floor.

Links and muted text use color to distinguish themselves from body. Color alone may sit below WCAG; a half-step weight bump carries the difference the hue can't.

no floor
Read the recipe below — the link is set in regular weight, so the desaturated color is doing all the work, and muted asides drift toward the background.
v1 — weight floor
Read the recipe below — the link runs at weight 500, so the desaturated color is enough, and muted asides stay legible.
Gap in v2 spec §7.1 names the floors (--v-link-weight-min, etc.) but defaults all of them to inherit. The shipping palette should set them; the spec needs to say so. 
/* tokens/color.css — set, don't just declare */
--v-link-weight-min:    500;
--v-accent-weight-min:  500;
--v-muted-weight-min:   500;

/* base.css — apply on the targets */
a        { color: var(--v-link);  font-weight: var(--v-link-weight-min, inherit); }
.v-muted { color: var(--v-muted); font-weight: var(--v-muted-weight-min, inherit); }
11

Icon stroke thickens with the text weight around it.

An icon set inside a label (weight 500) should look slightly heavier than the same icon inside body copy (weight 400). v1 binds stroke-width to --v-font-weight.

Gap in v2 spec §11 lists tools but doesn't mention an icon base style. v2 implementers leave Lucide's default stroke; icons feel uniformly thin against bold labels. 
/* base.css — applies to the global icon class */
.lucide, .icon {
  display: inline-block;
  height: 1em;
  width: 1em;
  flex-shrink: 0;
  vertical-align: text-bottom;
  stroke-width: clamp(2, var(--v-font-weight) / 200, 3);
}
12

Headings shift one step down inside <header>.

A card's title is visually an h3 but should be semantically an h2 inside that card's section. v1 lets authors pick the right semantic level by sizing headings smaller when wrapped in <header>.

Stand-alone <h2>

Without the rule, an <h2> inside a card heads is huge — authors compensate by using <h3>, breaking outline semantics.

Same <h2>, inside <header>

Sized to read as a card title. The author still writes the semantically correct level for the document outline.
Gap in v2 spec PLAN.md phase 1.5 ("base.css — headings, ...") doesn't carry forward v1's header h1, h2 { @mixin v-text-xl } auto-demotion. 
/* base.css */
h1          { font-size: var(--v-font-size-h1); line-height: var(--v-line-height-h1); }
header h1, h2 { font-size: var(--v-font-size-h2); line-height: var(--v-line-height-h2); }
header h2, h3 { font-size: var(--v-font-size-h3); line-height: var(--v-line-height-h3); }

IV

Spacing, density, rhythm

2 principles · maps to v2 spec §5.3 / §6
13

The spacing mixin moves border-width and radius too.

Denser layouts want thinner borders and tighter radii — not just smaller padding. v1's v-spacing-{sm,md,lg} moves all three; v2's spec moves only the pad/gap quintet.

sm · pad 0.5 · border 1 · r 0.25
md · pad 0.9375 · border 1 · r 0.4
lg · pad 1.5 · border 2 · r 0.75
Gap in v2 spec §5.3's v-spacing $size mixin sets only --v-spacing + the four pad/gap derivatives. v1's bundle is wider: --v-spacing, --v-border-width, --v-radius. 
/* tokens/spacing.css */
@define-mixin v-spacing $size, $border-width: 1px {
  --v-spacing:      $size;
  --v-pad-block:    $size;
  --v-pad-inline:   $size;
  --v-gap-block:    $size;
  --v-gap-inline:   calc($size * 0.5);
  --v-border-width: $border-width;          /* add */
  --v-radius:       calc($size - (1lh - 1em) / 2);  /* add */
}
14

Hi-dpi screens get a density bump automatically.

A 2.25rem control on a desktop reads fine; on a phone, fingers want at least 2.75rem. v1 ships a @media (resolution >= 200dpi) block that grows touch targets and bumps body text.

desktop · < 200dpi

Body text reads at 15px. Controls 30px high.

touch / hi-dpi · ≥ 200dpi

Body text 17px. Controls 44px to clear thumb-target.

Gap in v2 spec §5.1 names --v-input-height: 2.5rem as a single literal. No mention of the resolution-based density branch. PLAN.md phase 1 doesn't mention it either. 
/* tokens/spacing.css */
:root {
  --v-input-height: 2.25rem;
  --v-font-size:    0.9375rem;
}
/* Bigger touch targets + readable text on dense screens. */
@media (resolution >= 200dpi) {
  :root {
    --v-input-height: 2.75rem;
    --v-font-size:    1.0625rem;
    --v-spacing:      1rem;
  }
}
V

Context-aware chrome

6 principles · maps to v2 spec §10
15

Cards flatten when they're inside a container.

A card floating inside a popover or modal earns its shadow — it's a thing sitting above other things. The same card streaming down an o-container page is just a section break — shadows compete with body text.

inside `o-container`
Card on page No shadow. Sits in the reading flow.
inside a modal / popover
Card on backdrop Shadow, because it's floating.
Gap in v2 spec §10.1 says cards have a --v-shadow-outer. No mention of the conditional flattening inside o-container or restoring inside :modal / :popover-open. 
/* components/card.css */
.c-card {
  box-shadow: var(--v-shadow-outer);
}
@media (min-width: 50rem) {
  .o-container .c-card          { box-shadow: none; }
  :modal .c-card,
  :popover-open .c-card          { box-shadow: var(--v-shadow-outer); }
}
16

Inner shadows tell users when content scrolls.

When a modal body has more content above the fold, the top edge gets an inward shadow. Same for below. The chrome reports the affordance — no separate "more" badge.

scroll inside →

The cascade architecture sketch. Janus declares its layers up front and never lets a single class span more than one.

Reset normalizes defaults. Base styles elements directly. Components own chrome. Objects own structure. Variants set knobs. Tools win.

This is a deliberate inversion from v1, where utilities crept into components. The layer order is enforced by the cascade, not by hope.

The line between an object and a component is "structure vs. chrome." A box owns spacing and radius. A card adds shadow, border, and color on top.

The narrow tools layer (t-px-0, t-flex-fill, etc.) is intentionally tiny.

Top/bottom shadows fade as you reach each end.

Gap in v2 spec §10.2 mentions c-modal chrome but not the scroll-edge shadows. PLAN.md phase 5 lists scroll handlers under dom/handlers but the CSS spec for the shadows is missing. 
/* tokens/shadow.css */
--v-shadow-inner-top:    inset 0 6px 6px -4px rgb(0 0 0 / 15%);
--v-shadow-inner-bottom: inset 0 -6px 6px -4px rgb(0 0 0 / 15%);

/* components/modal.css — driven by dom handler that sets data-scroll-* attrs */
.c-modal__content:not([data-scroll-top]) {
  box-shadow: var(--v-shadow-inner-top);
}
.c-modal__content:not([data-scroll-bottom]) {
  box-shadow: var(--v-shadow-inner-bottom);
}
17

The tab indicator self-themes against whatever surface it sits on.

v1 colors the selected tab's border with the same color-mix recipe as a dynamic border, but mixing against the tone (primary / success / danger). Drop a tabs strip in any context; the indicator picks up the right color.

Gap in v2 spec §10.2 names c-tabs. No mention of the indicator-color recipe. The v-colors-* variant a tab strip sits inside should drive its selected color. 
/* components/tabs.css */
.c-tabs__tab[aria-selected="true"] {
  --v-border-color: color-mix(in hsl,
    var(--v-border-dynamic-base) var(--v-border-dynamic-mix),
    var(--v-accent));
  /* box-shadow used instead of border so neighboring tabs don't shift */
  box-shadow: inset 0 -1px 0 var(--v-border-color);
}
18

Disabled-but-clickable submit, when the form is invalid.

A "fully disabled" submit can't move focus to the invalid field on click — which is how screenreader users navigate to errors. v1 visually disables (opacity 0.5) but keeps pointer-events alive.

Enter a valid email address.

Click Save changes — focus moves to the invalid field, even though the button looks disabled.

Gap in v2 spec PLAN.md phase 6 mentions SubmitButton but doesn't carry forward v1's critical a11y detail: the submit must accept clicks even when visually disabled so it can dispatch focus. 
/* components/card.css */
.c-card:has(:invalid, [aria-invalid="true"]) .c-card__footer [type="submit"] {
  opacity: 0.5;
  cursor: not-allowed;
  /* deliberately NOT pointer-events: none —
     submit handler catches click and moves focus to first invalid field */
  pointer-events: unset;
}
19

Empty error containers collapse to zero height.

A field's error slot stays in the DOM for live-region announcements, but it shouldn't reserve a row of dead space when there's no error. v1 collapses with :has(... [data-t-empty]) + :empty.

Both fields ship with the same DOM. The second row reclaims the vertical space.

Gap in v2 spec §10.1 mentions c-alert. No mention of empty-collapse or data-t-empty sentinel from the form engine. Without it, forms develop dead vertical bands. 
/* components/alert.css */
.c-error-message:empty { display: none; }

.c-alert:has(.c-alert__children[data-t-empty]) { display: none; }

/* dom/handlers/t-empty.ts sets data-t-empty when there are no children */
20

Every text element ellipsizes by default — opt out, not in.

v1 sets overflow-x: hidden; text-overflow: ellipsis on the inline-text element set in base.css. A long heading in a narrow card never overflows; the consumer doesn't have to remember.

no truncation

A heading with a deliberately long string that pushes past the frame

v1 default

A heading with a deliberately long string that pushes past the frame

Gap in v2 spec §11 lists t-truncate as an opt-in single-line ellipsis. v1 ships truncation as the default in base.css for the inline-text set, with an explicit opt-out for prose containers. 
/* base.css */
a, aside, b, blockquote, em, h1, h2, h3, h4, h5, h6, p, span, strong {
  overflow-x: hidden;
  text-overflow: ellipsis;
}
/* … and explicit opt-out where you want wrapping */
.t-wrap { overflow-x: visible; text-overflow: clip; }

Drop-in additions to PLAN.md

Each row maps a principle to the PLAN.md phase where it belongs. Tokens go in Phase 1 (CSS foundation). Behaviors go in Phase 5 (DOM layer). The numbered cross-refs follow the design spec.

Phase 1.4 · tokens/color.css§5.1 · §7

Add --v-border-dynamic-base / --v-border-dynamic-mix / --v-border-color (computed). Add --v-body-bg (gradient stack). Split --v-ring / --v-ring-alt. Set palette-shipping defaults for --v-link-weight-min / --v-muted-weight-min / --v-accent-weight-min. Principles 01, 02, 07, 10.

Phase 1.4 · tokens/spacing.css§5.3

Extend the v-spacing $size mixin to bundle --v-border-width and --v-radius in addition to the four pad/gap derivatives. Add the @media (resolution >= 200dpi) density bump for --v-input-height, --v-font-size, --v-spacing. Principles 13, 14.

Phase 1.5 · base.css§5 · §11

Specify the a:focus-visible double-underline rule. Specify default truncation for the inline-text set (with explicit opt-out for prose). Specify icon stroke-width binding (.lucide { stroke-width: clamp(2, var(--v-font-weight) / 200, 3) }). Specify header h1, h2 { ... } auto-demotion. Principles 08, 11, 12, 20.

Phase 3a · variants/colors.css§7

Add v-colors-tooltip (always-inverted), v-colors-popover (matches card chrome), v-colors-code (tinted-paper), v-colors-pre (ink, even in light mode), v-colors-callout (auto weight-bump). These aren't tonal — they're surface-role variants and need to be enumerated alongside the tonal set. Principles 03, 04, 05.

Phase 3a · tools/states.css§11

Add the t-hover mixin (filter: contrast(0.95) saturate(1.15); will-change: filter) and the t-focus-ring mixin (stacks halo + crisp outline). The hover-color story for components defers to t-hover rather than per-palette knobs. Principles 06, 07.

Phase 3b · components/card.css§10.1

Specify the conditional shadow rule: cards inside an o-container on wide viewports lose --v-shadow-outer; cards inside :modal / :popover-open always show it. Plus the :has(:invalid) [type="submit"] disabled-but-clickable rule. Principles 15, 18.

Phase 3b · components/tabs.css§10.2

Specify the selected-indicator border-color recipe — same color-mix as the dynamic border but with --v-accent (or the surrounding v-colors-* tone) as the mix target. Use inset box-shadow for the underline so neighboring tabs don't shift on selection. Principle 17.

Phase 3b · components/modal.css§10.2

Specify the scroll-edge inner shadows (--v-shadow-inner-top, --v-shadow-inner-bottom), wired to the data-scroll-top / data-scroll-bottom sentinels set by the DOM handler. Document them as a chrome contract between the CSS and the handler. Principle 16.

Phase 5 · dom/handlers§12

Add t-empty.ts (sets data-t-empty on containers whose children render no content — drives empty-alert collapse) and t-kb-nav.ts (toggles body[data-v-kb-nav] on first Tab / arrow-key, removes on first mousedown). The keyboard-vs-mouse flag is a single boolean but everything from menu hover to active-descendant respects it. Principles 09, 19.

Phase 1.3 · reset.csscross-cutting

Add the * { border-color: transparent } rule from v1's reset. v1's o-input-box reserves space for a border on every interactive element so the box doesn't shift on focus or invalid state. Without the transparent default, the border space is reserved but invisible until needed. Cross-cutting; supports principles 07, 17, 18.

A note on the principle behind these principles Every item above is one line of CSS that encodes a relationship rather than a value. Borders aren't a color, they're 17.5% of the way to the ink. Hover isn't a darker shade, it's a contrast filter. The link weight isn't 500, it's "whatever the floor is." When v2 implementers reach for literal values, the system loses its self-tuning property — and every new palette becomes a manual tweaking session. The v2 spec already names a lot of knobs; it needs to also name the formulas that connect them.
